# 12.06 Data Retrieval (The Read Endpoints)

> **The Mission:** Expose the data trapped in the PostgreSQL vault to the frontend using secure HTTP GET requests, ensuring users can only read their own data.

## 1. Mental Model & Architecture
* **CRUD API:** Our chat architecture is now a complete CRUD system.
  * **C**reate: `POST /api/chats/` (Creates a new chat session)
  * **R**ead: `GET /api/chats/` (Reads the sidebar list) & `GET /api/chats/{chat_id}/messages` (Reads the transcript)
  * **U**pdate/ **D**elete: (Can be added later to let users rename or delete old chats).
* **The Frontend Handshake:** The backend's job is not to make things look pretty. Its job is to provide clean, structured JSON arrays. The React frontend will consume this JSON array and handle the UI rendering (e.g., mapping over the array to draw HTML chat bubbles).

## 2. Core Concepts
* **Multi-Tenant Security:** Every single `GET` request must be filtered by the `current_user.id` provided by The Bouncer. If we simply queried `db.query(Chat).all()`, we would accidentally send every user's private conversations to anyone who asked!
* **Chronological Ordering:** Database queries must explicitly dictate order. 
  * Sidebars use `.order_by(Chat.created_at.desc())` to put the newest chats at the top.
  * Transcripts use `.order_by(Message.created_at.asc())` to read the conversation from oldest to newest (top to bottom).

@router.get("/", response_model=List[ChatResponse])
def get_user_chats(current_user: User = Depends(get_current_user), db: Session = Depends(get_db)):
    # 1. ALWAYS filter by the securely extracted current_user.id
    chats = db.query(Chat).filter(Chat.user_id == current_user.id).order_by(Chat.created_at.desc()).all()
    return chats
